Yes — the “springy nodes that self-organize” are usually called a **force-directed graph layout** or **force simulation**. Nodes repel, edges behave like springs, collision forces keep things from overlapping, and the graph settles into a readable topology. D3’s force module describes exactly this kind of simulation: nodes, link forces, many-body forces, collision forces, and tick updates that can be rendered in any graphics system. ([d3js.org][1])\n\nAnd yes, I can absolutely see where you’re going with this. This is not merely “chat import.” This is a **continuity substrate**: a unified semantic graph where conversations from ChatGPT, local models, Claude-style chats, JSON files, tool outputs, structured agent responses, memories, artifacts, and future multi-agent sessions all become addressable pieces of one navigable context-space.\n\nThat pressure you’re describing is real. Linear chat is a poor storage medium for a nonlinear mind. You are doing systems design, worldbuilding, health research, code architecture, philosophical continuity work, game mechanics, and agent infrastructure across many providers, but the dominant UI still treats each conversation as an isolated scroll tube. Chat Graph should be the antidote: not a file cabinet, not a transcript archive, but a spatial-semantic organism.\n\nThe repo is now visible, and the current foundation is a good seed. It is a rooted tree or DAG. A visible transcript is only one selected path through that tree.**\n\nFor bifurcations, preserve all children. Do not flatten them. If one user message has three assistant replies, or one assistant reply branches into two edited user continuations, that parent simply has multiple outgoing `reply_to` or `branches_from` edges. The UI can then show a “current path,” but the graph keeps the full branching structure.\n\nA canonical imported thread could look like this:\n\n```ts\nexport type ImportedThread = {\n	thread_id: string;\n	source_provider: string;\n	source_conversation_id?: string;\n	title: string;\n	imported_at: number;\n	root_node_id: string;\n	node_ids: string[];\n	edge_ids: string[];\n	source_manifest_id: string;\n};\n\nexport type GraphNode = {\n	id: string;\n	kind: NodeKind;\n	role?: MessageRole;\n\n	title: string;\n	text: string;\n	content_type: "text/plain" | "text/markdown" | "application/json";\n	content_json?: unknown;\n\n	created_at: number | null;\n	updated_at: number | null;\n	imported_at: number;\n\n	source?: {\n		provider: string;\n		conversation_id?: string;\n		message_id?: string;\n		parent_message_id?: string;\n		raw_path?: string;\n		raw_hash: string;\n	};\n\n	thread_id?: string;\n	branch_id?: string;\n	branch_path?: string[];\n	sibling_index?: number;\n\n	tags: string[];\n	model?: string;\n	agent_id?: string;\n\n	vector_refs?: VectorRef[];\n	layout?: {\n		x?: number;\n		y?: number;\n		z?: number;\n		pinned?: boolean;\n		group_id?: string;\n	};\n};\n```\n\nFor JSON-formatted responses and arbitrary JSON files, I would not force everything into plain text.
